# Fixes 2026-04-09 — Version 2.3.2

## Sicherheit

### K1 — XSS/Code-Injection bei Credential-Injection behoben (`WebViewContainer.js`)

Etwa die Hälfte der Login-Handler nutzte unsichere String-Interpolation beim Befüllen von Passwortfeldern via `executeJavaScript`. Ein Passwort mit Sonderzeichen wie `"` oder `\n` konnte die JS-Ausführung brechen oder beliebigen Code im Webview ausführen.

Alle betroffenen Stellen (Outlook, Moodle, BBB, Handbook, Office, Nextcloud-ADFS, schul.cloud) nutzen jetzt durchgängig `JSON.stringify()`:

```js
// Vorher (unsicher):
`document.querySelector('#passwordInput').value = "${password}";`

// Nachher (sicher):
`document.querySelector('#passwordInput').value = ${JSON.stringify(password)};`
```

## Datenkorrektheit

### K2 — Datenkorruption in `reencryptData` behoben (`DatabaseService.js`)

Die Methode zum Neu-Verschlüsseln beim Passwort-Wechsel enthielt zwei kritische Fehler:

1. `this.encryptionKey` wurde innerhalb der Schleife nach dem ersten Todo auf das neue Passwort gesetzt — alle weiteren Todos wurden danach mit dem falschen Schlüssel entschlüsselt und korrupt gespeichert.
2. Der `COMMIT` wurde ausgeführt, bevor die asynchronen `db.all()`-Callbacks ihre UPDATE-Statements abgeschlossen hatten.

Die neue Implementierung:
- Liest alle Daten **vor** der Transaction ein
- Entschlüsselt alle Einträge mit `oldPassword` im Speicher (CryptoJS ist synchron)
- Verschlüsselt danach alles mit `newPassword`
- Führt alle UPDATEs in einer einzigen awaited Transaction aus
- Setzt `this.encryptionKey` erst nach erfolgreichem `COMMIT`

### K9 — `createTables` resolved jetzt erst nach Abschluss aller DDL-Statements (`DatabaseService.js`)

`resolve()` wurde bisher synchron am Ende von `db.serialize()` aufgerufen, bevor die `db.run()`-Callbacks abgeschlossen waren. Fehler bei der Tabellenerstellung wurden nie gefangen. Jetzt erhält jedes `db.run()` einen Fehler-Callback; `resolve()` wird erst im Callback des letzten Statements aufgerufen.

## Logikfehler

### K3 — `||` statt `&&` bei Theme-Update (`electron.js`)

```js
// Vorher: zerstörte Fenster (die nicht mainWindow sind) passierten den Check
if (!win.isDestroyed() || win !== mainWindow) {

// Nachher: korrekt
if (!win.isDestroyed() && win !== mainWindow) {
```

Der Fehler führte zu einem Crash beim Theme-Wechsel, wenn ein zerstörtes Kind-Fenster in der Fensterliste war.

## Bugs

### K5 — Passwort-Toggle-Bug in den Einstellungen behoben (`SettingsPanel.js`)

WebUntis- und Schulportal-Passwortfelder nutzten beide den State `showBBBPassword`, sodass alle drei Felder gleichzeitig ein-/ausgeblendet wurden. Jetzt hat jedes Feld einen eigenen Toggle-State: `showBBBPassword`, `showWebuntisPassword`, `showSchulportalPassword`.

### K8 — Kontextmenü in Webviews war nicht funktionsfähig (`preload.js`)

Der IPC-Channel `showContextMenu` fehlte in der Whitelist des Preload-Skripts — die Nachrichten wurden still verworfen. `showContextMenu` ist jetzt in der Whitelist eingetragen.

## Memory Leaks

### K6 — Event-Listener-Leak bei Secure Files behoben (`electron.js`)

Jeder Aufruf von `open-secure-file` registrierte einen neuen `before-quit`-Handler via `app.on(...)`, der nie entfernt wurde. Bei vielen geöffneten Dateien häuften sich Handler an.

Lösung: Zentrales `secureFileCleanups` Set; ein einzelner `before-quit`-Handler iteriert darüber und räumt auf.

### K7 — Notification-Intervall-Leak behoben (`WebViewContainer.js`)

Das `dom-ready`-Event feuert bei jedem Seitenload. Jedes Mal wurde ein neues `setInterval` für die Benachrichtigungsprüfung erstellt, aber nur das letzte in der Ref gespeichert — alle vorherigen liefen endlos weiter. Jetzt wird das vorhandene Intervall vor jedem neuen `setInterval` erst geclearet.

## Weitere Fixes

### W4 — `autoUpdater.updateDownloaded` existiert nicht (`electron.js`)

`electron-updater` stellt keine `updateDownloaded`-Property bereit — der quit-and-install-Code beim App-Beenden war damit toter Code. Ein lokales `updateDownloaded`-Flag wird jetzt im `update-downloaded`-Event gesetzt.

### W5 — Null-Checks für autoUpdater-Events (`electron.js`)

Alle autoUpdater-Event-Handler prüfen jetzt ob `mainWindow` existiert und nicht zerstört ist, bevor sie `webContents.send()` aufrufen. Gemeinsamer `sendUpdateStatus()`-Helper eingeführt.

### W6 — `will-download`-Handler wurde mehrfach registriert (`electron.js`)

Da alle Webviews dieselbe Session (`persist:main`) teilen, wurde für jede neue WebContents-Instanz ein weiterer `will-download`-Handler auf derselben Session registriert — der Download-Dialog erschien mehrfach. Ein `downloadHandlerSessions` WeakSet verhindert jetzt Doppelregistrierungen.

### W9 — `off()`/`offMessage()` in Preload-Skript als No-Op markiert (`preload.js`)

`on()` wrapt den Callback in eine interne `subscription`-Closure. `off()` versuchte den Original-Callback zu entfernen und fand ihn nie. Beide Methoden sind jetzt dokumentierte No-Ops; zum Entfernen von Listenern sollte die von `on()` zurückgegebene Cleanup-Funktion verwendet werden.

### W14 — Unbenutzte Imports entfernt (`App.js`)

Sechs nie genutzte Imports aus `errorHandler.js` entfernt: `setupGlobalErrorHandling`, `handleCredentialError`, `handleWebViewError`, `safeExecute`, `performanceMonitor`, `ERROR_CATEGORIES`.

### W15 — `Date.now()` als ID durch `crypto.randomUUID()` ersetzt (`TodoList.js`, `SettingsPanel.js`)

Schnelles Hinzufügen von Einträgen konnte identische IDs erzeugen, was zu Schlüssel-Kollisionen und falschen Operationen (Löschen/Bearbeiten des falschen Eintrags) führen konnte.
